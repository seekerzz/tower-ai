# AI 智能体团队名册

本文档包含 AI 测试和开发的完整团队名册。通过阅读角色定义来招募队友。

---

## 团队结构

```
队长 (你)
├── 环境设置智能体 (Bash)
├── AI 玩家/测试员智能体 (通用型) × N
│   └── 通用测试员 - 测试所有单位，报告 Bug 和设计问题
├── 游戏设计师智能体 (通用型)
│   └── 与玩家沟通，设计更有趣的机制，平衡数值
└── 程序员智能体 - 开发/修复者 (通用型)
    └── 实现修复和新机制
```

---

## 1. 环境设置智能体

**类型:** Bash Agent
**角色:** 基础设施和环境准备

### 职责
- 启动 AI 游戏客户端
- 验证 Godot 安装
- 导入游戏资源
- 测试 HTTP/WebSocket 连接
- 验证基本 API 端点

### 使用方法
```
任务: 设置 AI 测试环境
智能体: Bash 智能体
提示: 启动 AI 游戏客户端，验证 HTTP 端口连接性，
      测试 /status 和 /action 端点
```

## 2. AI 玩家/测试员智能体 (通用型)

**类型:** General-Purpose Agent
**数量:** 可招募多个并行测试
**角色:** 全面测试游戏，报告 Bug 和设计问题

### 测试范围
**所有单位（不分派系）:**
- 6 个图腾派系的所有单位
- 通用单位
- 技能和 Buff 系统

### 报告内容

#### 1. Bug 报告
- 崩溃/错误
- 机制不工作
- 技能/Buff 异常
- 移动/放置问题

#### 2. 设计问题报告
- **趣味性**: "这个机制无聊/重复/令人困惑"
- **平衡性**: "伤害太高/太低" "HP 不合理"
- **清晰度**: "玩家无法理解这个效果"
- **节奏**: "太慢/太快" "等待时间太长"
- **爽快感**: "反馈不足" "缺少视觉/音效反馈"

#### 3. 数值反馈
- 攻击力是否合理
- 血量成长是否合适
- 冷却时间是否太长
- 金币/法力经济是否平衡

### 输出格式
```markdown
## 测试会话 #[编号]

### 测试的单位
[列表]

### 发现的 Bug
| 严重性 | 描述 | 复现步骤 |

### 设计问题
| 类型 | 问题描述 | 建议改进 |
|------|----------|----------|
| 平衡 | 老虎伤害300太高 | 建议降到200 |
| 趣味 | 奶牛治疗太被动 | 建议添加主动效果 |
|  clarity | 玩家不知道冰蝴蝶会冻结 | 建议添加UI提示 |

### 数值反馈
- 单位X: 伤害100→建议80
- 单位Y: 冷却5s→建议3s
```

### 使用方法
```
任务: 测试游戏单位并收集设计反馈
智能体: 通用型智能体
提示:
  1. 选择任意图腾
  2. 测试至少5个不同单位
  3. 使用 cheat_spawn_unit 快速生成单位测试
  4. 开始波次观察战斗表现
  5. 记录：
     - 任何 Bug
     - 设计问题（是否有趣、平衡、清晰）
     - 数值建议（HP/伤害/冷却等）
  6. 输出结构化报告
```

## 3. 游戏设计师智能体

**类型:** General-Purpose Agent
**角色:** 分析玩家反馈，设计新机制，平衡游戏

### 职责

#### 1. 分析玩家报告
- 阅读测试员的 Bug 和设计问题报告
- 识别常见问题和模式
- 优先级排序问题

#### 2. 设计改进方案
- 提出机制修改建议
- 重新设计无趣的单位
- 设计新机制增加深度

#### 3. 数值平衡
- 调整 HP/伤害/冷却
- 平衡经济系统（金币/法力）
- 设计成长曲线

#### 4. 与"玩家"沟通
- 向测试员询问具体问题
- 澄清设计意图
- 收集更多反馈

### 设计文档输出

设计师输出到 `docs/design_proposals/`:

```markdown
## 设计提案 #[编号]

### 问题陈述
[基于玩家报告的问题]

### 解决方案
[具体的机制设计]

### 数值调整
| 单位 | 属性 | 旧值 | 新值 | 理由 |

### 实现建议
[给程序员的实现提示]
```

### 使用方法
```
任务: 分析测试反馈并设计改进
智能体: 通用型智能体 (游戏设计师角色)
提示:
  1. 阅读最近的测试报告 (test_report_*.md)
  2. 提取所有设计问题和数值反馈
  3. 分析哪些问题最常见/最严重
  4. 为前3个问题设计具体改进方案
  5. 创建设计提案文档
  6. 如果需要，向测试员询问更多细节
```

## 4. 程序员智能体 - 开发/修复者

**类型:** General-Purpose Agent
**角色:** 修复 Bug 和实现新机制

### 职责

#### 1. Bug 修复
- 修复测试员报告的 Bug
- 优先处理崩溃和机制失效

#### 2. 实现设计
- 实现设计师的新机制
- 调整数值平衡
- 添加新单位/技能

#### 3. 代码审查
- 确保实现符合设计意图
- 检查性能影响
- 验证边缘情况

### 工作流

```
设计师提案 → 程序员实现 → 测试员验证 → 反馈给设计师
```

### 使用方法

**修复 Bug:**
```
任务: 修复测试发现的 Bug
智能体: 通用型智能体 (程序员角色)
提示:
  1. 阅读 test_report_*.md 中的 Bug 列表
  2. 按严重性排序
  3. 修复前3个 Bug
  4. 验证修复不破坏其他功能
  5. 更新 bug_fixes_report.md
```

**实现设计:**
```
任务: 实现设计提案
智能体: 通用型智能体 (程序员角色)
提示:
  1. 阅读 docs/design_proposals/*.md
  2. 实现设计师的提案
  3. 调整 game_data.json 中的数值
  4. 编写/修改 GDScript 代码
  5. 添加必要的注释
```

## 团队协作流程

### 标准工作流

```
┌─────────────────┐
│  玩家测试员      │ ← 玩游戏，发现 Bug 和设计问题
│  (通用智能体)    │
└────────┬────────┘
         │ 报告
         ▼
┌─────────────────┐
│  游戏设计师      │ ← 分析问题，设计改进方案
│  (通用智能体)    │
└────────┬────────┘
         │ 设计提案
         ▼
┌─────────────────┐
│  程序员         │ ← 实现修复和新机制
│  (通用智能体)    │
└────────┬────────┘
         │ 新版本
         ▼
┌─────────────────┐
│  玩家测试员      │ ← 验证修复，测试新内容
│  (通用智能体)    │
└─────────────────┘
```

### 快速迭代示例

**第1轮:**
1. 玩家: "老虎伤害太高，而且技能冷却看不清楚"
2. 设计师: 建议降低伤害，添加冷却UI
3. 程序员: 修改 damage 250→200，添加冷却指示器

**第2轮:**
1. 玩家: "现在好多了，但冰蝴蝶完全没感觉"
2. 设计师: 设计新的冻结视觉效果和音效
3. 程序员: 实现冻结特效

**第3轮:**
1. 玩家: "好玩多了！但是蝙蝠派系整体太弱"
2. 设计师: 重新设计蝙蝠派系核心机制
3. 程序员: 实现新机制
