## 执行摘要

通过对多份崩溃日志的黑盒分析，我发现了 CRASH-002 的关键规律。该崩溃具有以下特征：

1. **崩溃时机绝对固定**: 100% 发生在第1波战斗正式开始时
2. **错误信息一致**: `ERROR: Parameter "t" is null.`
3. **触发条件明确**: 与图腾攻击定时器在波次切换时的状态竞争有关
4. **影响范围**: 所有图腾流派均受影响

### 关键发现

**所有崩溃的共同点**:
1. 崩溃都发生在 `"【波次事件】第 1 波战斗正式开始！"` 日志之后
2. 崩溃时都有单位已部署在战场上
3. 错误信息完全一致: `Parameter "t" is null`
4. 崩溃后游戏进程继续，但战斗机制未实际执行

### 时间窗口分析

从 "第1波战斗正式开始" 到崩溃的时间间隔：
- 牛图腾日志1: 几乎立即崩溃（<1秒）
- 牛图腾日志2: 几乎立即崩溃（<1秒）
- 蝙蝠图腾日志: 约0.6秒 (03:525 → 04:130)
- 狼图腾日志: 未记录精确时间戳

**结论**: 崩溃发生在波次开始后的极短时间内，与波次初始化逻辑强相关。

## Taunt/嘲讽机制关联分析

### 牛图腾机制回顾

根据 GameDesign.md，牛图腾具有以下特性：

| 机制 | 描述 | 触发条件 |
|-----|------|---------|
| 受伤充能 | 受到伤害时积累能量 | 核心受击时 |
| 全屏反击 | 每5秒触发一次全屏攻击 | 能量满或定时触发 |
| 嘲讽联动 | 牦牛守护可吸引敌人攻击 | 每5秒触发 |

### 牦牛守护 (Yak Guardian) 技能分析

从 GameDesign.md 提取：
- **LV.1**: 每隔5s吸引周围的敌人攻击自己
- **LV.2**: 每隔4s吸引周围的敌人攻击自己
- **LV.3**: 图腾反击时牦牛攻击范围内敌人额外受到牦牛血量20%伤害

**关键观察**: 牦牛守护的嘲讽技能也是每5秒触发一次！

### 定时器冲突假说

我推测崩溃可能与以下定时器冲突有关：

1. **图腾攻击定时器**: 所有图腾都有5秒攻击间隔
2. **波次开始初始化**: 第1波开始时需要初始化敌人
3. **单位技能定时器**: 如牦牛守护的嘲讽每5秒触发

**时间线冲突场景**:
```
T=0: 第1波战斗开始信号发出
T=0: 图腾攻击定时器触发（可能恰好在波次开始时）
T=0: 尝试获取敌人列表进行攻击
T=0: 敌人列表为空或包含null节点
T=0: 崩溃：Parameter "t" is null
```

值得注意的是，之前狼图腾的崩溃（CRASH-WOLF-001）已被定位并修复。对比两个崩溃：

| 错误信息 | Parameter "t" is null | Parameter "t" is null |
| 发生时机 | 第1波开始时 | 第1波开始时 |
| 根本原因 | 缺少 `is_wave_active` 检查 | 待确定 |
| 修复状态 | 已修复 | 三轮修复未成功 |
| 影响范围 | 仅狼图腾 | 所有图腾 |

**关键差异**: CRASH-WOLF-001 是代码级问题（缺少状态检查），而 CRASH-002 在修复后仍然出现，说明可能存在更深层的时序或竞争条件问题。

## 技术总监调查建议

基于黑盒分析，我提出以下调查方向：

### 建议1: 检查图腾攻击定时器的波次状态判断

**优先级**: P0

问题: 即使添加了 `is_wave_active` 检查，是否可能在波次开始瞬间出现竞争条件？

建议检查:
```
- GameManager.is_wave_active 的 setter 是否有延迟？
- 定时器触发和波次开始信号是否存在时序竞争？
- 是否需要在获取敌人列表前进行双重检查？
```

### 建议2: 检查敌人列表的初始化时机

**优先级**: P0

问题: 波次开始时敌人是否已完全初始化？

建议检查:
```
- 敌人 spawn 和加入活跃列表是否是原子操作？
- get_nearest_enemies() 是否可能返回包含 null 的数组？
- 敌人节点的 is_instance_valid() 检查是否充分？
```

### 建议3: 检查牦牛守护嘲讽技能的初始化

**优先级**: P1

问题: 牦牛守护的嘲讽定时器是否在部署时立即启动？

建议检查:
```
- YakGuardian 的 _ready() 或初始化逻辑
- 嘲讽定时器是否在单位部署时立即触发
- 嘲讽目标选择是否处理了敌人列表为空的情况
```

### 建议4: 添加更详细的日志埋点

**优先级**: P1

为便于后续黑盒分析，建议在以下位置添加日志：

```
- 波次开始前的状态快照（敌人数量、图腾定时器状态）
- 图腾攻击触发时的详细日志（目标数量、目标有效性）
- 敌人列表变化时的日志（添加/移除敌人）
- 定时器触发和波次状态的关联日志
```

### 建议5: 检查 Godot 引擎版本相关问题

**优先级**: P2

问题: 这是否是 Godot 引擎特定版本的问题？

建议检查:
```
- Timer 节点在场景切换时的行为
- 信号连接在节点释放时的处理
- 数组操作中的 null 值处理
```

## 黑盒观察到的异常现象

### 异常1: 日志格式不一致

不同日志文件的时间戳格式不一致：
- 新日志: 无时间戳前缀，格式为 `【状态同步】...`
- 旧日志: 带时间戳 `[2026-03-02 08:24:25.064] [OBSERVATION] [OBS] ...`

**影响**: 增加了日志分析的难度，建议统一日志格式。

### 异常2: 崩溃后测试继续执行

在 `ai_session_cow_totem_20260302_082425.log` 中：
- 08:24:35.841 发生崩溃
- 08:24:45.712 测试继续执行 "步骤5: 购买牦牛守护"

这表明崩溃被捕获后游戏继续运行，但战斗机制可能已损坏。

### 异常3: 单位购买意图与实际不符

在 `ai_session_cow_totem_20260302_082425.log` 中：
- AI意图: 购买 iron_turtle
- 实际购买: spiderling

这可能表明商店系统存在状态同步问题。

## 结论与建议

### 核心结论

1. **CRASH-002 是一个时序相关的竞争条件问题**，而非简单的空指针检查遗漏
2. **崩溃与波次开始时的初始化逻辑强相关**，特别是图腾攻击定时器和敌人列表初始化的时序
3. **三轮修复未成功** 表明问题可能比表面看起来更复杂，可能涉及 Godot 引擎的定时器机制

### 对测试阻塞的影响评估

| 被阻塞的测试 | 优先级 | 建议处理方式 |
|-------------|-------|-------------|
| 牛图腾机制验证 | P0 | 等待CRASH-002修复 |
| 蝙蝠图腾机制验证 | P0 | 等待CRASH-002修复 |
| 蝴蝶图腾机制验证 | P1 | 等待CRASH-002修复 |
| 毒蛇图腾机制验证 | P1 | 等待CRASH-002修复 |
| 鹰图腾机制验证 | P1 | 等待CRASH-002修复 |

### 给技术总监的紧急建议

1. **在 MechanicTotemBase 或各图腾实现中添加防御性编程**：
   - 在获取敌人列表后，遍历前检查每个元素的有效性
   - 在 deal_damage 调用前添加 null 检查
   - 考虑使用 Godot 的 `is_instance_valid()` 函数

2. **检查波次状态切换的原子性**：
   - 确保 `is_wave_active = true` 和敌人初始化是原子操作
   - 或者考虑在波次开始信号发出前完成所有敌人初始化

3. **考虑延迟图腾攻击定时器的启动**：
   - 在波次开始后的短暂延迟（如0.5秒）后再启动图腾攻击定时器
   - 确保敌人已完全初始化并可被攻击
